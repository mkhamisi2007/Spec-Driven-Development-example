# Implementation Plan: Company Website

**Branch**: `001-company-website` | **Date**: 2026-06-21 | **Spec**: [spec.md](spec.md)

**Input**: Static website for software company with About, Products (real-time search), and Partners pages. Vanilla HTML/CSS/JS, JSON data files, client-side filtering, no backend.

## Summary

Build a fully static, client-side website for a software company consisting of three pages (About, Products, Partners) with responsive design. The Products page includes real-time search filtering powered by vanilla JavaScript. All data is stored in external JSON files and loaded via Fetch API. The site requires no backend, no build tools, and no framework dependencies—pure HTML5, CSS3, and ES6+ JavaScript deployable to GitHub Pages or Netlify.

## Technical Context

**Language/Version**: Vanilla JavaScript (ES6+), HTML5, CSS3 — no TypeScript, no transpilers

**Primary Dependencies**: None (no frameworks, no build tools, no npm packages)

**Storage**: JSON files (products.json, partners.json) in `/data` directory; loaded via Fetch API

**Testing**: Manual browser testing + automated accessibility checks (axe DevTools or WAVE)

**Target Platform**: Web browsers (modern Chrome, Firefox, Safari, Edge — latest versions)

**Project Type**: Static website / Client-side single-page application (SPA-like with page navigation)

**Performance Goals**: 
- Initial page load <2 seconds on 4G networks
- Product search filter <200ms response time
- Lighthouse Performance & Accessibility scores ≥90

**Constraints**: 
- No server-side code; static files only
- Page size budget: HTML <50KB, CSS <30KB, JS <50KB (gzip)
- Must work offline after initial load (all assets cached)
- Responsive 320px–1920px (mobile-first design)
- WCAG 2.1 AA accessibility compliance

**Scale/Scope**: 
- Products: 5–10 sample, up to 500 production
- Partners: 3–5 sample
- Pages: 3 (About, Products, Partners)
- User journeys: Browse company info, search products, view partners, cross-page navigation

## Phase 0: Research Tasks

### Research Findings

1. **Real-time client-side search performance**
   - Decision: Use native JavaScript `Array.filter()` on products array
   - Rationale: Fast enough for 500 items; no library dependency needed
   - Performance: Filter + DOM update completes in <50ms for typical datasets
   - Alternative rejected: MiniSearch/Lunr would add 10–15KB for marginal benefit

2. **JSON data loading pattern**
   - Decision: Fetch with error handling and fallback to empty array
   - Rationale: Standard pattern, works in all modern browsers, explicit error handling
   - Code: Single `data-loader.js` module to handle both products.json and partners.json
   - Alternative rejected: ES6 import for JSON only works in modules; fetch() is simpler

3. **Mobile-first responsive CSS**
   - Decision: CSS Grid/Flexbox with 768px media query breakpoint
   - Rationale: No dependencies, universally supported, aligns with Constitution
   - Mobile layout (<768px): Single-column, hamburger menu, stacked navigation
   - Desktop layout (≥768px): Multi-column products, always-visible horizontal nav
   - Alternative rejected: Tailwind/Bootstrap add 30–50KB unnecessary weight

4. **Accessible hamburger menu**
   - Decision: Toggle class on `<nav>` element controlled by button with ARIA attributes
   - Rationale: Simple (5–10 lines of JavaScript), fully accessible, no dependencies
   - Keyboard support: Enter/Space to toggle, Escape to close
   - ARIA: `aria-expanded`, `aria-controls`, `aria-label` for accessibility

5. **Lighthouse performance optimization**
   - Decision: Image optimization (WebP + PNG fallback), CSS critical path, deferred JS
   - Image optimization: Partner logos as WebP ~30% smaller than PNG
   - CSS strategy: Main styles inline for above-fold, defer component styles
   - JavaScript: Load search functionality only on products page
   - Target: 90+ Performance and Accessibility scores achievable with vanilla stack

**Conclusion**: Vanilla HTML/CSS/JS approach is optimal. All research confirms that no additional libraries, frameworks, or build tools are needed. The performance targets are easily achievable with proper optimization.

## Phase 1: Design & Contracts

### 1. Data Model

**Products** (data/products.json)

```typescript
interface Product {
  id: string;              // Unique identifier (e.g., "prod-001")
  name: string;            // Product name (required)
  description: string;     // Product description (required)
  category: string;        // Category (e.g., "CRM", "Analytics", "Security")
  pricing: string;         // Pricing (e.g., "$99/month" or "$50–500/month")
}

interface ProductsFile {
  products: Product[];
}
```

Example:
```json
{
  "products": [
    {
      "id": "prod-001",
      "name": "CloudSync Pro",
      "description": "Real-time data synchronization across cloud platforms",
      "category": "Integration",
      "pricing": "$299/month"
    },
    {
      "id": "prod-002",
      "name": "SecureVault",
      "description": "Enterprise-grade encryption and secure credential storage",
      "category": "Security",
      "pricing": "$199/month"
    }
  ]
}
```

**Partners** (data/partners.json)

```typescript
interface Partner {
  id: string;              // Unique identifier (e.g., "partner-001")
  name: string;            // Partner name (required)
  logo: string;            // Logo image URL/path (required; e.g., "/images/partners/logo.png")
  description: string;     // Partner description (optional)
}

interface PartnersFile {
  partners: Partner[];
}
```

Example:
```json
{
  "partners": [
    {
      "id": "partner-001",
      "name": "Acme Solutions",
      "logo": "/images/partners/acme-logo.png",
      "description": "Leading enterprise integration platform"
    }
  ]
}
```

**Validation Rules**:
- Product: All fields (id, name, description, category, pricing) must be non-empty strings
- Partner: id, name, logo must be non-empty; description is optional
- IDs must be unique within their respective arrays
- Pricing format is flexible (user-defined string)
- Logo path must reference a valid image file

### 2. Interface Contracts

**Data API (Fetch)**

GET `/data/products.json`
- Response: 200 OK with JSON matching ProductsFile schema
- Failure handling: If fetch fails, log error and display "Unable to load products"; use empty array fallback
- Caching: Cache in browser localStorage with timestamp for offline access

GET `/data/partners.json`
- Response: 200 OK with JSON matching PartnersFile schema
- Failure handling: If fetch fails, display "Unable to load partners"; use empty array fallback
- Caching: Cache in browser localStorage with timestamp for offline access

**Search Filter Function (Client-Side)**

```javascript
function filterProducts(query, products) {
  if (!query.trim()) return products;
  const lowerQuery = query.toLowerCase();
  return products.filter(p => p.name.toLowerCase().includes(lowerQuery));
}
```
- Input: Search string (can be empty), products array
- Output: Filtered products array (case-insensitive substring match on product name)
- Performance: <200ms for 500 products
- Behavior: Returns all products if query is empty; returns empty array if no matches

**Navigation Function (Client-Side)**

```javascript
function navigateTo(page) {
  // Show/hide page content
  // Update navigation menu active state
  // Update window.location.hash for browser history
  // Load data if not already cached
}
```
- Input: Page identifier ("about", "products", "partners")
- Output: Page displayed, navigation updated
- Side effects: URL fragment updated (#about, #products, #partners)
- Performance: Immediate (no loading delay for cached data)

**Validation Scenarios**

1. **All three pages load with correct content**
   - Action: Navigate to index.html in browser

   - Action: Click "Products" in navigation menu
   - Expected: Products page loads, product list displays
   - Action: Click "Partners" in navigation menu

2. **Product search filters in real-time**
   - Action: Go to Products page; type "cloud" in search box

   - Action: Clear search box
   - Expected: All products displayed again
   - Action: Search for non-existent product (e.g., "xyz123")
   - Expected: "No products found" message displayed

3. **Mobile navigation works correctly**
   - Setup: Open site on mobile device (iPhone, Android) OR simulate with Chrome DevTools Device Emulation
   - Action: Verify hamburger menu icon visible on screens <768px

   - Expected: Navigation menu reveals with About, Products, Partners links
   - Action: Tap "Products" link
   - Expected: Products page loads, menu collapses
   - Action: Verify "Products" is highlighted as active in navigation

4. **Images and assets load correctly**
   - Action: Open browser DevTools → Network tab
   - Action: Reload page
   - Expected: No 404 errors for images, CSS, or JavaScript files
   - Action: All partner logos display correctly without broken image placeholders

5. **Lighthouse performance audit passes**
   - Action: Open Chrome DevTools → Lighthouse
   - Action: Run audit on each page (About, Products, Partners)
   - Expected: Performance score ≥90, Accessibility score ≥90
   - Expected: No Critical issues reported

6. **Keyboard navigation and accessibility**
   - Action: Tab through page — navigate with keyboard only (no mouse)
   - Expected: All links and interactive elements receive focus
   - Action: Click hamburger menu with keyboard (Tab + Enter)
   - Expected: Menu opens/closes with keyboard
   - Action: Run axe DevTools or WAVE extension
   - Expected: No Critical accessibility violations
   - Action: Test with screen reader (VoiceOver/NVDA)
   - Expected: Page structure and content are readable by screen reader
