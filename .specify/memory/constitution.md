# Software Store Constitution

## Core Principles

### I. Simplicity First

Every feature must be as simple as possible to implement, maintain, and use. No unnecessary abstractions, libraries, or complexity. Prefer vanilla HTML, CSS, and JavaScript over frameworks and build tools when feasible. Complexity must be justified and documented.

**Why**: Static sites succeed through clarity and directness. Overengineering increases maintenance burden, deployment fragility, and page load times. Simplicity is a feature.

### II. Static Architecture (No Backend, No Database)

The entire application is a static website. No server-side logic, no database, no authentication backends. All dynamic behavior is client-side JavaScript. Data is served via static files (JSON, CSV, or embedded in HTML). Site must function perfectly without any server infrastructure beyond static file hosting.

**Why**: Static sites are inherently secure, infinitely scalable, and trivially deployable. This constraint forces clarity in what the application truly needs.

### III. Responsive & Clean UI

Every interface must be responsive and work flawlessly on mobile, tablet, and desktop. UI must be clean, minimal, and focused. No decorative bloat. Every visual element must serve a purpose. Follow web design fundamentals: hierarchy, whitespace, contrast, legibility.

**Why**: Users access from any device. Clean design reduces cognitive load and improves usability. Every unnecessary element slows the page and confuses the user.

### IV. Web Accessibility (WCAG 2.1 AA Minimum)

All pages MUST meet WCAG 2.1 AA accessibility standards. Semantic HTML required. Keyboard navigation required. Color contrast MUST meet WCAG standards. Screen reader compatibility mandatory. Alt text for all images. Form labels and error messages must be clear and associated correctly.

**Why**: Accessibility is not optional. It is a legal, ethical, and practical requirement. Accessible sites are faster, more maintainable, and serve more users. Inaccessible sites will fail review.

### V. Performance & Fast Page Load

Target: Initial page load <2 seconds, fully interactive <3 seconds on 4G networks. Images must be optimized (WebP with fallbacks). CSS and JS must be minified. No render-blocking resources. Lazy load images and deferred scripts. Measure with Lighthouse and real-world metrics (Core Web Vitals).

**Why**: Users abandon slow sites. Search rankings depend on performance. Network-constrained users cannot afford waste. Performance is a design constraint, not an afterthought.

### VI. Maintainable HTML, CSS, and JavaScript

Code MUST be readable, well-organized, and self-documenting. HTML must be semantic and structured logically. CSS must use consistent naming, avoid deep nesting, minimize global scope. JavaScript must be modular, use clear function names, avoid inline logic in HTML. No minification in source; generated by build process only. Every file must have a clear purpose.

**Why**: Single developer or team must be able to understand and modify any part of the site in minutes, not hours. Maintenance cost dominates over the long term. Bad code becomes technical debt that slows all future work.

## Technical Standards

### Allowed Technologies
- **HTML5**: Semantic markup only (no deprecated elements)
- **CSS3**: Vanilla CSS preferred; CSS Grid and Flexbox for layout
- **JavaScript**: Vanilla ES6+ (no jQuery or large frameworks); small focused libraries OK (e.g., Alpine.js, htmx) if justified
- **Data**: JSON files, CSV, or embedded in HTML
- **Hosting**: Any static host (Netlify, Vercel, GitHub Pages, AWS S3, etc.)
- **Build**: Minimal tooling (npm scripts, Gulp, 11ty, Hugo for templating only)

### Prohibited Technologies
- Backend frameworks (Node.js, Python, Ruby, Django, Rails, etc.)
- Databases (SQL, NoSQL, CMS, headless CMS)
- Authentication services (Auth0, Firebase, etc.)
- API backends for dynamic data
- Heavy frameworks (React, Vue, Angular, Next.js, etc.)
- CSS preprocessors (Sass, LESS) unless output is maintainable vanilla CSS

### Performance Budget
- HTML: <50 KB per page (gzip)
- CSS: <30 KB total (gzip)
- JS: <50 KB total (gzip)
- Images: Optimized; WebP format preferred with JPG/PNG fallback
- Fonts: Maximum 2 typeface families; subset glyphs

## Development Practices

### Code Organization
- Every component (reusable HTML block) lives in its own file or clearly marked section
- CSS organized by logical component, not by file type
- JavaScript modules: one responsibility per file
- Naming: Use descriptive, semantic names (avoid `container-1`, `util-fix`, etc.)

### Testing & Validation
- Automated accessibility checks in CI (axe, WAVE, or similar)
- Manual testing on real devices (iOS Safari, Chrome, Firefox, Edge)
- Lighthouse audit required before any deployment: target score 90+ on Performance, Accessibility, Best Practices
- Link validation (no broken internal or external links)
- Mobile-first testing (test on mobile first, then expand to desktop)

### Deployment & Maintenance
- Every page must deploy independently (no shared state beyond static assets)
- Version static assets with content hashing to enable aggressive caching
- Monitor 404s and redirect loops in production
- Update CSS framework and library dependencies quarterly; batch updates
- Security: Regular scan for vulnerable dependencies (npm audit)

### Code Review Standards
- All changes reviewed before merge
- Review checklist: Is this the simplest solution? Is performance affected? Is accessibility broken? Is the code readable?
- Complexity justified in code comments when unavoidable
- Accessibility and performance regressions block merge

## Governance

This constitution is the source of truth for all development decisions. Deviations must be explicitly justified and documented in the PR.

**Amendment Process**: Constitution changes require discussion and documented rationale. Version increments follow semantic versioning: MAJOR for principle changes, MINOR for new guidance, PATCH for clarifications.

**Compliance Review**: Every PR must verify alignment with constitution principles. Automated checks (Lighthouse, axe) gate merge; manual accessibility review required.
